# Technical Specification

---

## 1. General

- **Platform:** iOS only
- **Framework:** Flutter (Dart)
- **Widget library:** Cupertino only ‚Äî do not use Material widgets
- **UI language:** English only ‚Äî all labels, buttons, text, and messages must be in English
- **Network:** fully offline ‚Äî no internet, no push notifications, no background services, no home screen widgets
- **Distribution:** App Store

---

## 2. Before You Start

1. **Colors:** check `additional info/color_scheme.css`.
   - If it has color values ‚Äî use them as the app's color palette.
   - If it is empty or you are told to use temporary colors ‚Äî pick a fitting palette, write all chosen colors into `color_scheme.css` using CSS custom property format (e.g. `--color-background: #0D0D1A;`). This way the client can later send updated colors in the same format and you simply replace the values, then update the Flutter `AppColors` class accordingly.

## 3. Project Structure

- `main.dart` must be nearly empty ‚Äî it only calls `runApp()` and launches the main screen. No logic, no configuration here.
- Organize code into logical folders: `screens/`, `widgets/`, `models/`, `services/`, `theme/`, `utils/`.
- Each screen lives in its own file.

---

## 4. Implementation Requirements

- **Fully implement** every screen ‚Äî no placeholder screens, no "coming soon" stubs for UI.
- You may add features and improvements beyond what is specified if they make the app better, more polished, or more unique. Use your judgment.
- Adaptive design is required ‚Äî the app must look and work well on iPhone (all sizes) and iPad. Use `MediaQuery` for responsive sizing.
- All text must be readable at all times. Ensure sufficient contrast. If text sits on a complex or dark background, add a text shadow or stroke to maintain legibility.

---

## 5. Screens

Implement all screens required by the concept. At minimum:

- **Main screen** (`main_screen.dart` or equivalent entry-point file) ‚Äî contains the root navigation (bottom navbar or tab bar)
- **Welcome / Splash screen** ‚Äî see Section 6
- **Achievements screen** ‚Äî dedicated full screen, not a modal or a tab inside another screen
- **Trophies screen** (if trophies exist in the app) ‚Äî dedicated full screen

Navigation must use a **custom, visually polished bottom navigation bar** (Cupertino-style). The navbar must be animated ‚Äî selected icons/tabs should have a visible animated state (glow, pulse, scale, or similar).

---

## 6. Welcome / Splash Screen

- Shown **every time the app launches** ‚Äî not just the first time.
- One single screen, no pages or scroll ‚Äî everything fits on one view.
- Contains: app logo/name, and a short description of what the user will find inside the app.
- **Auto-dismisses after 5 seconds** using a fade-out animation.
- **Tapping anywhere on the screen** also dismisses it ‚Äî the same fade-out animation must still play (do not skip it on tap).
- After the fade-out completes, the main screen of the app appears.

---

## 7. Animations

**Everything must be animated.** This is a hard requirement, not optional polish.

Specifically:
- Backgrounds must be animated (particles, drifting elements, shimmer, moving gradients, etc.)
- Icons must be animated (entrance animation, idle animation, or interaction animation)
- Navbar icons must be animated (selected state must be visually animated)
- All panels and cards must have entrance animations (staggered fade-in, slide-up, etc.)
- All screen transitions must be animated (custom page route transitions ‚Äî not the default push)
- Every interactive element (buttons, selectors, list items) must respond visually to tap (e.g. scale down on press, spring back on release)
- Use emoji where it fits and animate them where possible
- XP bars, progress bars, counters ‚Äî all must animate when their value changes

Use Flutter's built-in animation tools (`AnimationController`, `TweenAnimationBuilder`, `AnimatedContainer`, `AnimatedOpacity`, etc.). You may add `flutter_animate` or similar packages if they simplify implementation.

---

## 8. Gamification

Add gamification elements. You have full creative freedom here ‚Äî add whatever you think will make the app engaging. Required minimum:

- **XP system** ‚Äî users earn XP for actions in the app
- **Rank / level system** ‚Äî progression based on XP with named ranks
- **Streaks** ‚Äî track consecutive days of activity
- **Achievements** ‚Äî at least 15 achievements with a dedicated screen (see Section 5)
- **Trophies** ‚Äî awarded for milestones, displayed on a dedicated screen (see Section 5)
- Rank-up must trigger a **full-screen celebration animation** (particles, badge zoom, etc.)
- Achievement unlock must trigger an **animated toast/banner** (slides in from top, auto-dismisses)

---

## 9. AppsFlyer Event Stubs

Create `lib/services/appsflyer_service.dart`. The SDK will be integrated later. For now, all methods must:
- Print to debug console using `debugPrint`
- Be named clearly after the event they represent
- Accept relevant parameters as a `Map<String, dynamic>`

Cover all key user actions: app launch, core feature usage, achievement unlocked, rank-up, and any other meaningful events. Write a comment `// TODO: replace with real AppsFlyer SDK call` in each method.

---

## 10. Code Quality

- Minimum **6,000 lines of Dart code** across all files ‚Äî implement fully, do not cut corners
- No hardcoded strings in widgets ‚Äî use constants or named variables
- All data models must have `toJson()` / `fromJson()` for `shared_preferences` serialization
- Use `shared_preferences` for all local persistence

---

## 11. Git Commits

After completing each meaningful section of work, make a **local git commit** of everything done so far.  
Do **not** push to any remote repository ‚Äî local commits only.

Use conventional commit format:
```
git add .
git commit -m "feat: <short description of what was done>"
```

Commit after each screen, after setting up the theme, after implementing gamification, etc. Keep commits granular and descriptive.
